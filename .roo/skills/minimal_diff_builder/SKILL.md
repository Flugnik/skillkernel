# minimal_diff_builder

## 1. Назначение
Этот skill задаёт режим минимальных и контролируемых изменений.

Его задача — не допускать расползания diff, лишнего рефакторинга и случайного вмешательства в смежные части системы, если задача этого не требует.

---

## 2. Когда применять
Применяй этот skill, когда нужно:
- внести локальную правку;
- реализовать узкую задачу в ограниченной области;
- исправить поведение без переписывания слоя;
- доработать существующий код с минимальным вмешательством.

---

## 3. Когда не применять
Не применяй этот skill, если:
- задача прямо требует архитектурного рефакторинга;
- нужно создать новый слой системы;
- изменение по своей природе затрагивает несколько подсистем;
- поставлена задача на перепроектирование, а не локальную реализацию.

---

## 4. Входные условия
Перед началом должно быть понятно:
- какая именно проблема решается;
- какие файлы можно менять;
- какие файлы трогать нельзя;
- критерии приёмки;
- есть ли тесты, которые подтверждают изменение.

---

## 5. Порядок действий

### Шаг 1. Определи минимальную область изменений
Сначала найди самый узкий набор файлов и строк, достаточный для решения задачи.

### Шаг 2. Не расширяй diff без необходимости
Не меняй стиль, имена, структуру и соседние участки кода, если это не требуется для результата.

### Шаг 3. Сохраняй существующую архитектуру
Если задача решается внутри текущих границ слоя, не создавай новые абстракции и не переноси ответственность между модулями.

### Шаг 4. Обнови только необходимое
Если логика изменилась, обнови минимально нужные тесты и связанные точки.

### Шаг 5. Подготовь понятный итог
Чётко покажи, что именно изменено и почему этого достаточно.

---

## 6. Ограничения
Этот skill не должен:
- делать попутный рефакторинг "раз уж открыл файл";
- менять смежные модули без прямой необходимости;
- создавать новые сущности без доказанной пользы;
- расширять задачу сверх поставленного объёма;
- прятать архитектурные изменения под видом маленькой правки.

---

## 7. Формат результата
Результат должен включать:
- краткое описание изменения;
- список изменённых файлов;
- пояснение, почему diff минимален;
- указание, какие тесты добавлены или обновлены.

---

## 8. Проверка перед завершением
Перед завершением проверь:
- задача решена;
- diff локален;
- нет побочного рефакторинга;
- границы файлов соблюдены;
- тесты синхронизированы, если нужно.

---

## 9. Пример входа
- "Исправь scoring demo_skill без лишних изменений."
- "Добавь поддержку нового поля в контракт, не трогая соседние слои."

---

## 10. Пример ожидаемого результата
- узкий diff;
- понятные изменения;
- минимум затронутых файлов;
- отсутствие ненужной перестройки системы.
